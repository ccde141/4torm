# 信风 · 工作流

信风用于将 Agent、会议和人工操作组织成可以保存并重复运行的可视化工作流。你可以在画布上放置不同类型的节点，再通过连线决定任务、上下文和结果如何向后传递。

每个工作流拥有独立的共享工作区。各 Agent 节点保留自己的执行过程，同时可以通过上游传来的信封继续处理任务，并在同一目录中读写文件。

节点调用内置文件工具时，会自动应用所绑定 Agent 的执行权限；工作流共享目录始终作为当前工作区纳入允许范围。

同一张工作流既可以手动运行，也可以自动运行。手动模式允许人类进入会议、与 Agent 对话或在暂停点作出决定；自动模式则面向无人值守执行，并可以按照循环档案重复运行。

## 搭建工作流

### 节点类型

信风提供六种节点。节点负责不同阶段的工作，普通连线则决定信封如何在它们之间传递。

| 节点 | 作用 | 可使用的运行方式 |
|---|---|---|
| **入口** | 提供初始任务或背景，并将其交给下游。工作流至少需要一个入口。 | 手动、自动 |
| **Agent** | 绑定控制台中的一个 Agent，接收上游信封后使用自身模型、工具和技能完成任务。 | 手动、自动 |
| **会议室** | 让选定的 Agent 节点参与讨论，由会长在结束时生成纪要并交给下游。 | 仅手动 |
| **暂停点** | 收到信封后暂停流程，等待人类检查或修改内容，再继续交给下游。 | 仅手动 |
| **Note** | 向相连的 Agent 节点注入长期指引，不参与实际运行，也不会传递工作结果。 | 手动、自动 |
| **出口** | 收集最终信封，结束本次执行并生成运行输出与对话归档。每个工作流只能有一个出口。 | 手动、自动 |

Agent 节点绑定的是控制台中长期保存的 Agent。节点可以另外设置职责和投递延迟，但不会复制出新的 Agent 实体。

Note 只能连接到 Agent。未连接的 Note 可以作为画布备注保留，但不会影响任何节点。

### 画布操作

从左侧节点栏将节点拖入画布，或者在画布空白处点击右键并选择节点类型。节点可以自由移动和调整大小；右下角的小地图与缩放控件用于浏览较大的工作流。

点击节点后，右侧会打开配置面板。所有节点都可以修改显示名称和人类备注；不同类型的节点还会提供各自的配置项。人类备注只用于画布识别，不会进入 Agent 的提示词。

从节点右侧的连接点拖向另一个节点左侧，即可建立连线。在节点上点击右键，可以编辑、克隆或删除节点；在连线上点击右键，可以选择“删除连线”。删除节点时，与它相连的连线也会一并移除。

### 信封如何流转

从节点右侧的连接点拖向另一个节点左侧，即可建立连线。信风使用两种连线：

- **普通连线**用于传递工作信封。节点完成当前任务后，会将交接内容发送给所有直接相连的下游节点。
- **Note 连线**用于向 Agent 注入提示。它显示为虚线，只能从 Note 连接到 Agent，不会传递工作结果或触发节点运行。

一个节点存在多条普通入线时，会等待当前一轮的上游信封全部到达后再开始处理。普通连线不能形成环；入口不能连接上游，出口也不能继续连接下游。

信封是节点之间的交接内容，通常包含上游整理出的结果、必要背景和后续说明。它不等于上游 Agent 的完整会话历史；完整对话保留在原节点的运行记录中，下游只接收已经整理好的交接信息。

多个下游节点可以收到同一份信封。不同节点仍保留各自的对话上下文，但可以通过工作流共享目录读取和修改同一批文件。

## 保存和管理工作流

点击顶部保存按钮，可以保存当前工作流。页面也会每五分钟自动保存一次；运行工作流前，系统会先完成保存，确保本次运行使用的是画布上的最新内容。

双击顶部名称可以修改工作流的显示名称。名称下方的工作流 ID 是其稳定标识，不会随重命名改变，工作区和历史运行记录仍会归属于同一个工作流。工作流名称仅用于显示；重命名后，工作流目录、共享工作区、循环档案和历史运行记录均保持原位置不变。

左侧工作流列表用于新建、加载和删除工作流。删除工作流会同时清除它的画布定义、共享工作区、循环档案和历史运行记录。当前工作流仍在运行时不能删除，需要先停止执行。

信风执行期间不提供全局工具注册，无论手动运行还是自动运行，Agent 节点都只使用启动时已经配置的工具和技能。工作流需要新工具时，应在运行前通过季风会话或气旋工位私聊完成创建与确认。

## 运行方式

同一张工作流可以选择手动运行或自动运行。两种方式使用相同的节点和连线，但对人类介入、节点类型和模型能力的要求不同。同一时刻只能运行一个信风工作流。

### 手动运行

点击顶部的「手动运行」，工作流会从入口开始执行一遍。工作流运行期间，可以随时打开任意 Agent 节点查看对话；节点空闲时，也可以直接发送消息，不必等待上游信封到达。

Agent 节点会依次处理收到的信封。运行到会议室或暂停点时，可以由人类进入对应面板参与讨论、补充信息或决定何时继续。最终信封到达出口后，本次运行结束。

### 自动运行

点击「自动运行」会打开循环档案面板。选择「不循环 · 单圈自动运行」时，工作流会在无人介入的情况下执行一遍；也可以创建循环档案，让同一张工作流连续运行多圈。

循环档案可以设置圈间等待时间、运行圈数和循环备注，并决定如何把上一圈的结果交给下一圈：

- **累积**：将上一圈的完整产出带入下一圈。
- **重置**：下一圈重新从初始任务和循环备注开始。
- **摘要**：先压缩上一圈的产出，再将摘要带入下一圈。

每一圈都会创建独立的执行记录和节点会话，但继续使用同一个工作流共享目录。圈间等待从上一圈完成后开始计算；选择永续运行时，工作流会持续执行，直到人为停止或某一圈运行失败。

自动运行不支持会议室和暂停点。

### 启动校验

开始运行前，系统会先保存当前画布并检查工作流结构，防止错误的节点配置或连接关系进入执行。如有问题，工作流不会启动，页面会列出需要修改的内容。

自动运行还会检查 Agent 使用的模型是否支持原生工具调用。

## 查看和介入执行

工作流运行后，画布会直接反映节点状态。蓝色脉冲表示节点正在处理内容，外侧的琥珀色光环表示它正在处理上游信封。Agent 正常完成当前一轮工作时，会短暂显示约 15 秒的绿色反馈；Agent 被手动停止或发生错误时不会显示绿色，而会在对应节点上保留红色停止或错误状态，方便定位中断位置。暂停点等待人类处理时也会单独闪烁。

右侧的「工作流 · 进度」侧板汇总各 Agent 节点的状态、当前信封和交接产出。正常完成、停止和报错会分别保留对应结果，不会把被中断的节点显示为已完成。工作流结束后，侧板会保留本次结果并转为只读；循环运行时会显示当前圈数。

### Agent 节点

工作流运行期间，点击 Agent 节点上的「交流」可以打开它的独立会话。模型输出、思考过程、工具调用、SubAgent 委托和节点间联络都会在其中显示。

节点空闲时，可以直接发送消息；节点正在处理信封或其他消息时，仍可查看执行过程，但需要等待当前处理结束后才能继续发言。

人类主动发起的回复可以直接停止。处理工作流信封时，则可以暂停当前节点；暂停会保留尚未交付的信封，不会把半完成结果发送给下游，之后可以选择续跑。

### 会议室节点

会议室收到上游信封后开始运行。进入会议面板，可以查看参与者的依次发言、思考过程和工具调用，也可以在公共讨论中补充信息，或通过右侧分列的会长私聊对话框与会长交流。

会议进行期间，可以增加、移除或调整参与者顺序。点击「结束会议」后，公共讨论会被锁定，会长生成会议纪要并将其作为信封交给下游。

### 暂停点

信封到达暂停点后，工作流会停在该节点等待处理。点击节点上的「编辑」，可以检查或修改信封内容；选择「继续」后，确认过的内容才会发送给下游。

暂停点只会阻塞当前依赖链，不会取消已经在其他分支中运行的节点。

## 停止工作流

工作流运行时，顶部按钮会切换为「■ 停止」。点击后会停止整个工作流：

- 取消所有进行中的 LLM 调用和工具执行。
- 尚未完成的信封不会继续交给下游节点；已经完成并交接的内容会保留在本次运行记录中。
- 被中断的 Agent 节点保留红色停止状态，Agent 错误也保留红色错误状态；正常完成的节点保留已有产出，绿色反馈按正常时长显示。
- 运行侧板会保留本次结果并转为只读。停止是本次运行的终态，不能从中间位置继续；需要再次执行时重新启动工作流。

停止当前 Agent 回复和停止整个工作流不是同一个操作：

- 人类主动发起的回复可以点击「停止」，只中断当前回复，工作流本身仍可继续运行。
- 处理上游信封或 `contact` 联络时，不能只丢弃当前回复。需要暂缓处理时使用「暂停」，它会保留信封并允许之后续跑；需要结束整次执行时再使用「停止工作流」。

## 数据目录

信风把工作流定义、长期共享工作区和每次运行的归档分开保存。工作流 ID 是目录名，重命名只会修改显示名称，不会改变这些路径。

```
data/tradewind/
├── workflows/          ← 工作流定义和长期共享工作区
│   └── {id}/
│       ├── graph.json      ← 画布上的节点和边
│       ├── meta.json       ← 显示名称、更新时间
│       ├── profiles.json   ← 循环档案（创建循环档案后才会有）
│       └── workspace/      ← 该工作流所有节点共享的长期工作区
└── runs/               ← 每次执行的独立归档
    └── {wfId}/
        └── {execId}/
            ├── meta.json            ← 执行状态和开始/结束时间
            ├── graph-snapshot.json  ← 本次运行实际使用的画布快照
            ├── events.jsonl         ← 节点事件流水
            ├── output.json          ← 出口节点产出（运行到出口后生成）
            ├── meetings/            ← 会议室节点的轮次归档
            └── nodes/                ← 各 Agent 节点的运行记录
                └── {nodeId}/
                    └── messages.json
```

`workflows/{id}/workspace/` 会跨运行保留。同一工作流的不同轮次继续使用这份共享目录；删除工作流时，定义、共享工作区、循环档案和对应运行记录会一并删除。

`runs/{wfId}/{execId}/` 只对应一次执行。`meta.json` 的状态可能是 `running`、`done`、`error`、`stopped` 或 `crashed`；程序重新启动时，会把上次遗留的 `running` 记录标记为 `crashed`。部分文件只有在相应节点或功能实际运行后才会出现。

这些目录属于本地运行数据，已被 Git 忽略，不会随代码提交。Git 忽略不等于备份；需要迁移工作流、共享文件或历史运行记录时，请单独备份 `data/tradewind/`。
